Deep Research Agent

Runtime de investigacion profunda construido en Python y LangGraph para ejecutar planes iterativos con checkpoints humanos, persistencia por thread_id, salida JSON estable y empaquetado listo para CLI, Docker Compose y skill de repositorio.

Resumen operativo

Que cubre este repositorio

Implementado hoy
  • CLI con run, resume y list-pending.
  • Grafo LangGraph con planeamiento, busqueda, extraccion, reflexion y sintesis.
  • Persistencia de runs y checkpoints con SQLite o Postgres.
  • Servicios para OpenAI y Ollama, busqueda Tavily/Serper/fallback sin API, y reporte Markdown con citas.
  • Empaquetado con Docker Compose y skill deep-research-agent-compose.
Importante para integrar
  • No hay servidor HTTP ni servidor MCP listo para arrancar en este repo.
  • Los enums incluyen mas proveedores de los que el MVP soporta realmente.
  • firecrawl figura como enum, pero hoy falla explicitamente como backend de busqueda.
  • runtime/placeholders.py conserva nodos reservados de una fase previa y no participa del flujo real.

Arquitectura en capas

Interfaces arriba, runtime en el centro, servicios y persistencia debajo, y assets de entrega alrededor.

Contrato externo estable

La salida JSON de la CLI es la mejor superficie para skills, scripts de automatizacion y adaptadores MCP.

Control humano real

El agente siempre puede pausar con contexto estructurado antes de recuperar fuentes o antes de sintetizar.

Pruebas del comportamiento

La suite valida el loop completo, persistencia, formateo del reporte, configuracion y assets operativos.

            Configuracion y contratos

            Que se configura y que queda fijo

            Prefijo de entorno

            Toda la configuracion entra por variables DEEP_RESEARCH_*, con lectura desde .env.

            Backends de persistencia

            checkpoint_db_url deriva a SQLite o Postgres segun persistence_backend.

            Salida estable

            La CLI imprime JSON ordenado; esa salida es el contrato mas seguro para automatizar el agente.

            Guardas del loop

            max_iterations evita iteraciones infinitas y fuerza una revision de suficiencia.

            Modelos y proveedores de LLM
            • Soportados en el MVP: openai y ollama.
            • OpenAI requiere DEEP_RESEARCH_OPENAI_API_KEY.
            • Los enums incluyen anthropic, google y azure_openai, pero hoy fallan explicitamente si intentas usarlos.
            • El servicio exige respuestas JSON que validen contra esquemas Pydantic.
            Busqueda, extraccion y limites de contenido
            • none usa DuckDuckGo HTML y deduplicacion por URL canonica.
            • tavily y serper funcionan si hay API keys; si faltan, el sistema cae al fallback sin API.
            • firecrawl aparece como enum pero no esta soportado como backend de busqueda del MVP.
            • La extraccion no usa embeddings: toma texto bruto, corta por max_content_chars_per_source y selecciona pasajes por solapamiento de palabras clave.
            Persistencia, interrupciones y resultados
            • SQLite guarda el runtime localmente con aiosqlite y AsyncSqliteSaver.
            • Postgres requiere dependencias opcionales y se usa por defecto en Docker Compose.
            • Cada comando actualiza el RunStore con latest_state, latest_interrupts y last_message.
            • Un host externo solo necesita persistir o reenviar thread_id, decision humana y payload opcional.

            Integraciones

            Como llevarlo a otros proyectos

            Esta es la seccion util para acoplar el agente a otra herramienta. Skill y Docker Compose existen hoy. MCP es un patron de adaptacion recomendado sobre la CLI o sobre ResearchRuntimeService.

            Docker Compose

            Es la integracion operativa mas completa si quieres ejecutar el agente sin entorno Python local y con Postgres listo para persistir checkpoints.

            cp .env.example .env
./scripts/compose-agent.sh bootstrap
./scripts/compose-agent.sh run --query "Assess the AI infrastructure market"
./scripts/compose-agent.sh list-pending
./scripts/compose-agent.sh resume --thread-id <thread_id> --decision approve --summary "Proceed"
./scripts/compose-agent.sh down
            • Ventaja: el script espera a que Postgres este sano antes de ejecutar el comando real.
            • Contrato: el servicio app corre el entrypoint CLI y devuelve JSON por stdout.
            • Cuando usarlo: demos, operacion manual, skill de repositorio, integraciones en CI o hosts sin venv.

            Integracion Python en proceso

            Es la opcion correcta si tu host ya corre Python y quieres acoplar el agente sin shell ni contenedores.

            from deep_research_agent.domain.models import HumanDecisionType, ResearchRequest
from deep_research_agent.runtime.service import ResearchRuntimeService
from deep_research_agent.settings import AppSettings

settings = AppSettings()
service = ResearchRuntimeService(settings)

started = await service.start_run(
    ResearchRequest(query="Assess the AI infrastructure market"),
    max_iterations=4,
)

if started.resume_supported:
    completed = await service.resume_run(
        thread_id=started.thread_id,
        decision_type=HumanDecisionType.APPROVE,
        summary="Proceed with the current plan",
    )
            • Usa esta via si quieres reemplazar servicios, inyectar un bundle falso o envolver el runtime dentro de otra aplicacion.
            • Tambien es la mejor superficie para montar un servidor MCP sin shell intermedio.

            Adaptador MCP recomendado

            Este repo no trae servidor MCP, pero el runtime ya expone el contrato adecuado. Lo mas directo es publicar tres herramientas y mapearlas a la CLI o al servicio Python.

            Herramientas MCP sugeridas

research_run(
  query,
  audience="general",
  objective=None,
  constraints=[],
  deliverable_format="markdown",
  max_iterations=None,
)

research_list_pending(limit=20)

research_resume(
  thread_id,
  decision,
  summary="",
  payload={},
)
            • Si el host ya usa Python, llama a ResearchRuntimeService directamente.
            • Si el host esta aislado o corre en otro runtime, llama a la CLI o al wrapper de Compose y parsea el JSON.
            • Cuando la respuesta tenga pending_human_input, devuelvela al cliente MCP como un checkpoint visible, no como un error.
            • Preserva thread_id como identificador del hilo de investigacion y como clave de reanudacion.

            CLI local

            La CLI es la interfaz mas simple para scripts, wrappers propios o herramientas de build.

            deep-research-agent run --query "Assess LangGraph checkpointing"
deep-research-agent list-pending --limit 20
deep-research-agent resume --thread-id <thread_id> --decision approve --summary "Sufficient evidence"
            • Su salida JSON es estable y esta probada por la suite.
            • Es una buena base para un wrapper MCP, una extension o un job batch.

            Decisiones del MVP

            Decisiones técnicas tomadas

            JSON estructurado para LLM

            Planificacion, reflexion y sintesis validan contra esquemas Pydantic. Eso hace el runtime mas predecible y mas facil de integrar.

            Dos checkpoints humanos

            Se pausa antes de recuperar fuentes y antes de sintetizar. No es un agente completamente autonomo por diseño.

            Persistencia por thread_id

            Todo el flujo gira alrededor de un hilo reanudable. Esa misma clave simplifica skill, CLI, Compose y futuros adaptadores MCP.

            Compose con wrapper

            El script compose-agent.sh evita condiciones de carrera de docker compose run y concentra el bootstrap.

            Postgres por defecto en contenedores

            La experiencia containerizada prioriza persistencia durable. SQLite queda como backend ligero para local y tests.

            Busqueda tolerante a credenciales faltantes

            Si faltan API keys, Tavily y Serper vuelven al fallback sin API. La idea es mantener el runtime util incluso con configuracion minima.

            Heuristica simple de extraccion

            La extraccion usa HTML parser y overlap de keywords. Es suficiente para el MVP, pero es un candidato claro a mejora si integras fuentes internas o RAG.

            Limites explicitados, no escondidos

            Proveedores enumerados pero no soportados, firecrawl y runtime/placeholders.py se mantienen visibles para marcar que falta por implementar.

            Si vas a extender el proyecto, la ruta mas segura es mantener estable la superficie externa y cambiar por debajo los servicios o el grafo. Eso evita romper skills, wrappers de Compose y futuros adaptadores MCP.
